resource "aws_s3_bucket_public_access_block" "secure_bucket" {
  bucket = aws_s3_bucket.secure_bucket.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

resource "aws_s3_bucket_server_side_encryption_configuration" "secure_bucket" {
  bucket = aws_s3_bucket.secure_bucket.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "aws:kms"
    }
  }

"How do you manage cloud security vulnerabilities?"

Sample answer: "Cloud vulnerabilities are mostly misconfigurations rather than CVEs. I use CSPM tools like Defender for Cloud or Prisma Cloud for continuous monitoring of our cloud posture. In CI/CD, Checkov scans our Terraform before deployment to catch issues like public S3 buckets or overly permissive security groups. We also use AWS Config rules and Azure Policy for guardrails that prevent non-compliant deployments. The key difference from traditional VM is that we're checking configuration compliance, not patching software."

Summary

Cloud vulnerabilities are predominantly misconfigurations, not CVEs. Use CSPM tools for continuous monitoring, scan IaC in CI/CD, and implement guardrails via cloud-native policy services. The shared responsibility model means you own configuration security.
