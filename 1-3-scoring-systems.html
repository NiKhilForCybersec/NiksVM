This page provides a deep dive into vulnerability scoring systems including CVSS v3.1 and v4.0, EPSS (Exploit Prediction Scoring System), SSVC (Stakeholder-Specific Vulnerability Categorization), CISA KEV catalog, and techniques for building custom risk scoring formulas.

CVSS (Common Vulnerability Scoring System)

CVSS is the industry standard for assessing the severity of security vulnerabilities. Maintained by FIRST (Forum of Incident Response and Security Teams), CVSS provides a standardized framework for rating vulnerability severity from 0.0 to 10.0.

CVSS v3.1 Overview

CVSS v3.1, released in June 2019, is currently the most widely used version. It consists of three metric groups that progressively refine the score based on additional context.

Base Score Metrics

Base metrics represent intrinsic characteristics of a vulnerability that remain constant across time and environments:

Metric Values Description
Attack Vector (AV) Network (N), Adjacent (A), Local (L), Physical (P) How the vulnerability is exploited
Attack Complexity (AC) Low (L), High (H) Conditions beyond attacker's control
Privileges Required (PR) None (N), Low (L), High (H) Level of access needed to exploit
User Interaction (UI) None (N), Required (R) Whether user action is needed
Scope (S) Unchanged (U), Changed (C) Can exploit impact other components?
Confidentiality (C) None (N), Low (L), High (H) Impact on data confidentiality
Integrity (I) None (N), Low (L), High (H) Impact on data integrity
Availability (A) None (N), Low (L), High (H) Impact on system availability

Temporal Score Metrics

Temporal metrics adjust the Base score based on factors that change over time:

  • Exploit Code Maturity (E): Availability and sophistication of exploit code
  • Remediation Level (RL): Available fixes (official patch, workaround, etc.)
  • Report Confidence (RC): Confidence in the vulnerability's existence

Environmental Score Metrics

Environmental metrics allow organizations to customize scores based on their specific environment:

  • Modified Base Metrics: Adjust any base metric for local context
  • Confidentiality/Integrity/Availability Requirements: Business importance of CIA attributes

CVSS v4.0

CVSS v4.0, released in November 2023, introduces significant improvements addressing criticisms of previous versions. Key changes include:

  • New Nomenclature: CVSS-B (Base), CVSS-BT (Base+Threat), CVSS-BE (Base+Environmental), CVSS-BTE (all three)
  • Attack Requirements (AT): New metric capturing prerequisites like specific configurations
  • Improved Scope: Replaced with Vulnerable System (VC/VI/VA) and Subsequent System (SC/SI/SA) impacts
  • Supplemental Metrics: Additional context like Automatable, Recovery, Provider Urgency
  • Simplified Temporal â†’ Threat: Threat metrics now just Exploit Maturity

âš ï¸ CVSS Limitations

CVSS measures theoretical severity, not actual risk. A CVSS 10.0 on an isolated test server may pose less real risk than a CVSS 7.5 on an internet-facing production database. Always combine CVSS with business context and threat intelligence for prioritization.

EPSS (Exploit Prediction Scoring System)

EPSS, developed by FIRST, is a data-driven model that predicts the probability a vulnerability will be exploited in the wild within the next 30 days. Unlike CVSS which measures severity, EPSS measures likelihood of exploitation.

How EPSS Works

EPSS uses machine learning trained on historical exploitation data from multiple sources including threat intelligence feeds, exploit repositories, and honeypot data. The model considers factors like:

  • Vulnerability characteristics from CVE descriptions
  • Presence in exploit databases (Exploit-DB, Metasploit)
  • Social media and dark web mentions
  • Historical exploitation patterns for similar vulnerabilities
  • Time since disclosure

import requests
import json

def get_epss_scores(cve_list):
    """
    Fetch EPSS scores from FIRST API.
    Returns dict mapping CVE IDs to scores and percentiles.
    """
    base_url = "https://api.first.org/data/v1/epss"
    
    # API accepts comma-separated CVE list
    cve_param = ",".join(cve_list)
    
    response = requests.get(
        base_url,
        params={"cve": cve_param}
    )
    
    if response.status_code != 200:
        raise Exception(f"EPSS API error: {response.status_code}")
    
    data = response.json()
    
    results = {}
    for item in data.get('data', []):
        results[item['cve']] = {
            'score': float(item['epss']),
            'percentile': float(item['percentile']),
            'date': item['date']
        }
    
    return results

# Example usage
cves = ['CVE-2021-44228', 'CVE-2023-23397', 'CVE-2024-21762']
scores = get_epss_scores(cves)

for cve, data in scores.items():
    print(f"{cve}:")
    print(f"  EPSS Score: {data['score']*100:.2f}%")
    print(f"  Percentile: {data['percentile']*100:.1f}%")
    print()

SSVC (Stakeholder-Specific Vulnerability Categorization)

SSVC, developed by CISA and Carnegie Mellon's SEI, is a decision-tree approach to vulnerability prioritization. Unlike numeric scores, SSVC produces actionable recommendations: Track, Track*, Attend, or Act.

SSVC Decision Points

SSVC evaluates vulnerabilities across multiple decision points:

Exploitation Status

  • None: No evidence of exploitation
  • PoC: Proof-of-concept exists
  • Active: Actively exploited in the wild

Automatable

  • No: Requires manual steps, social engineering
  • Yes: Can be fully automated (wormable)

Technical Impact

  • Partial: Limited impact on CIA
  • Total: Complete system compromise possible

Mission Impact (for your organization)

  • None/Low: Minimal operational impact
  • Medium: Moderate disruption possible
  • High: Significant operational impact

SSVC Outcomes

Decision Action Timeline
Track Monitor, remediate per normal cycle Standard SLA
Track* Track with closer attention Within 30 days
Attend Prioritize, accelerated remediation Within 7 days
Act Immediate action, drop everything Within 24-48 hours

CISA KEV Catalog

The Known Exploited Vulnerabilities (KEV) catalog is maintained by CISA and lists vulnerabilities that have been actively exploited in the wild. This is a critical prioritization inputâ€”if a vulnerability is in KEV, it's not theoretical; attackers are actively using it.

KEV Criteria

To be added to KEV, a vulnerability must meet all criteria:

  • Has an assigned CVE ID
  • Evidence of active exploitation in the wild
  • Clear remediation guidance exists (patch or mitigation)

Using KEV for Prioritization

Python 
import requests
from datetime import datetime

def get_kev_catalog():
    """
    Download CISA KEV catalog and return as dict.
    """
    url = "https://www.cisa.gov/sites/default/files/feeds/known_exploited_vulnerabilities.json"
    
    response = requests.get(url)
    data = response.json()
    
    # Create lookup dict by CVE ID
    kev_dict = {}
    for vuln in data['vulnerabilities']:
        kev_dict[vuln['cveID']] = {
            'vendor': vuln['vendorProject'],
            'product': vuln['product'],
            'name': vuln['vulnerabilityName'],
            'date_added': vuln['dateAdded'],
            'due_date': vuln['dueDate'],
            'description': vuln['shortDescription'],
            'action': vuln['requiredAction']
        }
    
    return kev_dict

def check_kev_status(cve_list):
    """
    Check if CVEs are in KEV catalog.
    Returns list of dicts with KEV status.
    """
    kev = get_kev_catalog()
    
    results = []
    for cve in cve_list:
        if cve in kev:
            results.append({
                'cve': cve,
                'in_kev': True,
                'kev_data': kev[cve]
            })
        else:
            results.append({
                'cve': cve,
                'in_kev': False,
                'kev_data': None
            })
    
    return results

Building Custom Risk Scores

Organizations often need custom risk scoring that incorporates their specific business context. Here's a framework for building an effective custom scoring model:

Risk Score Components

Python 
class VulnerabilityRiskCalculator:
    """
    Custom risk scoring combining multiple factors.
    Produces score 0-100 where higher = more risk.
    """
    
    def __init__(self):
        # Weight factors (should total 1.0)
        self.weights = {
            'severity': 0.25,        # CVSS-based
            'exploitability': 0.25,   # EPSS + exploit status
            'exposure': 0.20,         # Network exposure
            'criticality': 0.20,      # Asset importance
            'threat_intel': 0.10      # KEV + active campaigns
        }
    
    def calculate_severity_score(self, cvss_score):
        """Normalize CVSS to 0-100 scale."""
        return (cvss_score / 10) * 100
    
    def calculate_exploitability_score(self, epss_score, exploit_status):
        """
        Combine EPSS probability with exploit maturity.
        epss_score: 0.0-1.0
        exploit_status: 'none', 'poc', 'weaponized', 'active'
        """
        # Base from EPSS (0-60 points)
        epss_points = epss_score * 60
        
        # Exploit maturity bonus (0-40 points)
        maturity_bonus = {
            'none': 0,
            'poc': 15,
            'weaponized': 30,
            'active': 40
        }.get(exploit_status, 0)
        
        return min(100, epss_points + maturity_bonus)
    
    def calculate_exposure_score(self, exposure_type, open_ports=None):
        """
        Score based on network exposure.
        exposure_type: 'internet', 'dmz', 'internal', 'isolated'
        """
        base_scores = {
            'internet': 100,
            'dmz': 75,
            'internal': 40,
            'isolated': 10
        }
        
        base = base_scores.get(exposure_type, 40)
        
        # Bonus for high-risk open ports
        risky_ports = {22, 23, 80, 443, 445, 3389, 1433, 3306, 5432}
        if open_ports:
            risky_open = len(set(open_ports) & risky_ports)
            base = min(100, base + (risky_open * 5))
        
        return base
    
    def calculate_criticality_score(self, asset_tier, data_classification):
        """
        Score based on asset importance.
        asset_tier: 'tier1', 'tier2', 'tier3', 'tier4'
        data_classification: 'public', 'internal', 'confidential', 'restricted'
        """
        tier_scores = {
            'tier1': 100,  # Crown jewels
            'tier2': 75,   # Business critical
            'tier3': 40,   # Standard production
            'tier4': 15    # Non-critical / test
        }
        
        data_multiplier = {
            'public': 0.5,
            'internal': 0.8,
            'confidential': 1.0,
            'restricted': 1.2
        }
        
        base = tier_scores.get(asset_tier, 40)
        multiplier = data_multiplier.get(data_classification, 0.8)
        
        return min(100, base * multiplier)
    
    def calculate_threat_intel_score(self, in_kev, active_campaign, industry_targeted):
        """
        Score based on threat intelligence.
        """
        score = 0
        
        if in_kev:
            score += 50  # CISA confirmed exploitation
        
        if active_campaign:
            score += 35  # Tied to active threat campaign
        
        if industry_targeted:
            score += 15  # Targeting our industry
        
        return min(100, score)
    
    def calculate_risk_score(self, vuln_data):
        """
        Calculate final weighted risk score.
        
        vuln_data = {
            'cvss': 9.8,
            'epss': 0.45,
            'exploit_status': 'weaponized',
            'exposure': 'internet',
            'open_ports': [443, 22],
            'asset_tier': 'tier1',
            'data_class': 'confidential',
            'in_kev': True,
            'active_campaign': True,
            'industry_targeted': False
        }
        """
        scores = {
            'severity': self.calculate_severity_score(
                vuln_data.get('cvss', 5.0)
            ),
            'exploitability': self.calculate_exploitability_score(
                vuln_data.get('epss', 0.01),
                vuln_data.get('exploit_status', 'none')
            ),
            'exposure': self.calculate_exposure_score(
                vuln_data.get('exposure', 'internal'),
                vuln_data.get('open_ports')
            ),
            'criticality': self.calculate_criticality_score(
                vuln_data.get('asset_tier', 'tier3'),
                vuln_data.get('data_class', 'internal')
            ),
            'threat_intel': self.calculate_threat_intel_score(
                vuln_data.get('in_kev', False),
                vuln_data.get('active_campaign', False),
                vuln_data.get('industry_targeted', False)
            )
        }
        
        # Calculate weighted total
        final_score = sum(
            scores[component] * self.weights[component]
            for component in scores
        )
        
        return {
            'final_score': round(final_score, 1),
            'component_scores': scores,
            'priority': self._get_priority(final_score)
        }
    
    def _get_priority(self, score):
        if score >= 80:
            return 'CRITICAL'
        elif score >= 60:
            return 'HIGH'
        elif score >= 40:
            return 'MEDIUM'
        else:
            return 'LOW'

    'cvss': 9.8,

    'data_class': 'confidential',
    'in_kev': True,
    'active_campaign': True,
    'industry_targeted': False
}

print(f"Risk Score: {result['final_score']}/100")
print(f"Priority: {result['priority']}")
print(f"Components: {result['component_scores']}")

Summary

Modern vulnerability management combines CVSS severity, EPSS probability, KEV exploitation status, and business context into risk-based prioritization.
