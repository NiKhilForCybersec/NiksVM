Zero-Day Response Procedures

How to respond when a new critical vulnerability drops with no patch available.

What is a Zero-Day?

A zero-day is a vulnerability that:
üÜï Is newly discovered (or newly public)
‚ö†Ô∏è Has NO patch available yet
üî• May already be exploited in the wild

Called "zero-day" because defenders have had ZERO DAYS to prepare!

Phase 1: DETECT (Hour 0-1)

‚ö° First Hour Actions
  • Monitor threat intel feeds (Twitter/X, vendor advisories, CISA)
  • Receive alert from security vendors, news, or internal detection
  • Verify the vulnerability is real (not a false alarm)
  • Determine if it affects your environment
  • Alert the incident response team
Key Intel Sources for Zero-Days
  • CISA Alerts: cisa.gov/uscert
  • Vendor Security Blogs: Microsoft, Cisco, VMware
  • Twitter/X: @CISAgov, @TheRecord_Media, @BleepinComputer
  • NVD: nvd.nist.gov
  • Your Scanner Vendor: Tenable, Qualys, Rapid7 alerts

Phase 2: IDENTIFY SCOPE (Hours 1-4)

üîç Scoping Questions
  • What's affected? Which products, versions, configurations?
  • How many assets? Run emergency scan or query CMDB
  • Where are they? Internet-facing? Internal? Cloud?
  • Who owns them? Identify asset owners for communication
  • What's the criticality? Crown jewels or standard assets?
Example: Log4Shell Scoping

When Log4Shell dropped, teams had to answer:

‚ùì "Which apps use Log4j?" ‚Üí Searched for log4j*.jar files
‚ùì "What versions?" ‚Üí 2.0-beta9 to 2.14.1 vulnerable
‚ùì "Internet-facing?" ‚Üí Web apps, APIs, load balancers
‚ùì "Can we detect exploitation?" ‚Üí Search logs for "${jndi:"

Many discovered Log4j in apps they didn't know had it!

Phase 3: CONTAIN (Hours 4-24)

Phase 4: ACT - Remediate (Days 1-7)

üîß Remediation Priority Order
  1. Internet-facing + Crown Jewels ‚Üí Fix within 24-48 hours
  2. Internet-facing + Standard ‚Üí Fix within 72 hours
  3. Internal + Crown Jewels ‚Üí Fix within 7 days
  4. Internal + Standard ‚Üí Fix within 14 days
When No Patch Exists
  • Apply vendor workarounds (registry keys, config changes)
  • Implement WAF rules to block exploit patterns
  • Enable enhanced monitoring/detection
  • Consider taking service offline if critical enough
  • Document risk acceptance if must remain online

Phase 5: SUSTAIN (Ongoing)

üìã Post-Zero-Day Actions
  • Monitor for patch release from vendor
  • Schedule patch deployment when available
  • Hunt for indicators of compromise (IOCs)
  • Update detection rules as new IOCs emerge
  • Conduct lessons learned / retrospective
  • Update runbooks based on experience

Interview Questions

üíº Common Interview Questions

Q: "How would you respond to a zero-day like Log4Shell?"

"I'd follow a structured response: First, DETECT - verify the vulnerability through trusted sources like CISA and vendor advisories. Second, IDENTIFY scope - query our asset inventory for affected products, prioritizing internet-facing and crown jewels. Third, CONTAIN - implement immediate mitigations like WAF rules blocking the JNDI pattern and removing the vulnerable JndiLookup class. Fourth, ACT - coordinate emergency patching starting with the highest-risk systems. Finally, SUSTAIN - hunt for IOCs, monitor for new exploit variants, and conduct a lessons learned."


Q: "What do you do when there's no patch?"

"Without a patch, I focus on compensating controls: network segmentation to limit exposure, WAF/IPS rules to block known exploit patterns, enhanced monitoring to detect exploitation attempts, and if necessary, taking the service offline temporarily. All decisions are documented with risk acceptance from appropriate stakeholders."
