Q: What is vulnerability management?
â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€
A: "Vulnerability management is a continuous process of identifying, 
evaluating, prioritizing, and remediating security weaknesses in 
systems and software. It includes asset discovery, vulnerability 
scanning, risk-based prioritization, remediation tracking, and 
verification. The goal is to reduce organizational risk by addressing 
exploitable vulnerabilities before attackers can leverage them."

Q: Explain the difference between vulnerability assessment and 
   penetration testing.
â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€
A: "Vulnerability assessment is automated scanning to identify known 
weaknessesâ€”it finds vulnerabilities but doesn't exploit them. Penetration 
testing is manual, authorized hacking that attempts to actually exploit 
vulnerabilities to demonstrate real-world impact. VA answers 'what 
vulnerabilities exist' while pentesting answers 'what can an attacker 
actually achieve.' VA is broad coverage; pentesting is deep validation."

Q: What is CVSS and how do you use it?
â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€
A: "CVSS (Common Vulnerability Scoring System) is a standardized way 
to assess vulnerability severity on a 0-10 scale. It has three metric 
groups: Base (intrinsic characteristics), Temporal (time-dependent 
factors), and Environmental (organization-specific context). I use 
CVSS as a starting point for prioritization but not the only factorâ€”
I also consider asset criticality, exposure, and threat intelligence 
like EPSS and CISA KEV status."

Q: What is the CISA KEV catalog?
â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€
A: "CISA KEV (Known Exploited Vulnerabilities) is a catalog maintained 
by CISA listing vulnerabilities that are actively being exploited in 
the wild. Federal agencies must remediate KEV vulnerabilities within 
specified timeframes. Even for non-federal organizations, KEV is a 
critical prioritization signalâ€”if it's in KEV, attackers are actively 
using it, so it should be top priority regardless of CVSS score."

Q: Walk me through your vulnerability management process.
â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€
A: "I follow a cyclical process:
1. Asset Discovery - Maintain current inventory of all systems
2. Scanning - Regular authenticated scans (weekly for critical, 
   monthly for standard systems)
3. Analysis - Review findings, validate severity, remove false positives
4. Prioritization - Rank by risk using CVSS, asset criticality, 
   EPSS/KEV, and business context
5. Remediation - Create tickets, assign owners, track against SLAs
6. Verification - Rescan to confirm fixes, close tickets
7. Reporting - Metrics to stakeholders, continuous improvement

The cycle runs continuously with different frequencies for different 
asset tiers."

Q: How do you prioritize vulnerabilities?
â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€
A: "I use risk-based prioritization combining multiple factors:
- CVSS score as baseline severity
- EPSS (Exploit Prediction Scoring) for likelihood of exploitation
- CISA KEV for known active exploitation
- Asset criticality (Tier 1-4 based on business impact)
- Network exposure (internet-facing vs internal)
- Compensating controls in place

A CVSS 7.0 on an internet-facing Tier 1 system with high EPSS gets 
higher priority than a CVSS 9.0 on an internal Tier 4 system with 
no public exploit."

Q: How do you handle a situation where a team can't meet the SLA?
â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€
A: "First, I understand the blockerâ€”is it technical, resource, or 
dependency-related? For legitimate delays, I work with them on 
compensating controls to reduce risk while waiting. This might be 
network segmentation, disabling the vulnerable feature, or enhanced 
monitoring. I document the exception with the compensating controls, 
get appropriate approval, and set a new remediation date. The key is 
reducing risk, not just meeting arbitrary deadlines."

Q: What's the difference between authenticated and unauthenticated scans?
â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€
A: "Unauthenticated scans only see what's externally visibleâ€”open ports, 
service banners, web pages. They miss internal vulnerabilities like 
missing patches or misconfigurations. Authenticated scans log into 
the system and check installed software versions, registry settings, 
file permissions, and configurations. Authenticated scans find 5-10x 
more vulnerabilities and are more accurate. I use authenticated scans 
for internal systems and unauthenticated for external perspective."

Q: How do you handle false positives?
â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€
A: "I investigate each suspected false positive by:
1. Reviewing the detection logic in the plugin/check
2. Manually verifying on the system
3. Checking if compensating controls make it non-exploitable
4. Researching vendor documentation

If confirmed false positive, I document the justification and either 
create a scanner exception for that specific instance or report it 
to the vendor if it's a detection bug. I never blanket-exclude 
vulnerabilitiesâ€”each exception is specific and documented."

A: "I'd start by:
1. Understanding the environmentâ€”network segments, asset types, 
   critical systems
2. Establishing scan credentials (service accounts with read access)
3. Creating asset groups by network segment and criticality
4. Defining scan policies appropriate for each group

7. Tuning to reduce false positives
8. Setting up recurring authenticated scans
9. Configuring reporting and alerting
10. Integrating with ticketing for remediation tracking"
