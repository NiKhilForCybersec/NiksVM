Scanner Types & Selection

ğŸ“– Section 2.1 â±ï¸ 15 min read ğŸ“… Last updated: 2025

This page covers the different types of vulnerability scanners including network-based, agent-based, and passive scanning approaches. You'll learn when to use each type, compare commercial vs. open source options, and understand key selection criteria.

Scanner Architecture Types

Vulnerability scanners come in several architectural flavors, each with distinct advantages and trade-offs. Understanding these differences is crucial for selecting the right tool and deployment model for your environment.

Network-Based Scanners

Network scanners operate from a central location and probe target systems over the network. They send specially crafted packets, analyze responses, and identify vulnerabilities based on service banners, version information, and behavioral analysis.

1. HOST DISCOVERY
   - ICMP ping, TCP SYN to common ports
   - ARP requests on local segments
   - Identify live hosts

2. PORT SCANNING
   - TCP connect/SYN scans
   - UDP service probes
   - Identify open ports and services

3. SERVICE ENUMERATION
   - Banner grabbing
   - Version detection
   - Service fingerprinting

4. VULNERABILITY DETECTION
   - Version-based checks (known vulnerable versions)
   - Behavioral checks (send probes, analyze responses)
   - Configuration checks (weak settings, defaults)

5. REPORTING
   - Aggregate findings
   - Assign severity scores
   - Generate remediation guidance

Network Scanner Advantages

  • No software installation required on targets
  • Can scan any IP-reachable system regardless of OS
  • Discovers unknown/unmanaged assets
  • Tests actual network exposure
  • Works across different environments from a single point

Network Scanner Limitations

  • Limited visibility without credentials (surface-level checks only)
  • Firewalls and network segmentation can block scans
  • Generates network traffic that may impact performance
  • Cannot assess systems behind NAT or in isolated networks
  • Point-in-time assessment (misses intermittent issues)

Agent-Based Scanners

Agent-based scanners deploy lightweight software agents on target systems. These agents continuously collect system information and report to a central management console, enabling real-time vulnerability visibility without network-based probing.

Agent Advantages

  • Deep visibility into system configuration and installed software
  • Works regardless of network location (cloud, VPN, roaming)
  • No network traffic for scanning (only reporting)
  • Continuous monitoring vs. point-in-time scans
  • Better detection of local privilege escalation vulnerabilities
  • Works through firewalls (outbound HTTPS only)

Agent Limitations

  • Requires agent deployment and management
  • Agent software must be compatible with target OS
  • Adds software footprint to systems
  • Cannot discover unknown assets (must be deployed first)
  • May have performance impact on endpoints
  • Requires agent health monitoring

Passive Scanners

Passive scanners monitor network traffic without actively probing targets. They analyze packet captures to identify services, versions, and potential vulnerabilities based on observed traffic patterns.

Passive Scanner Use Cases

  • OT/ICS environments where active scanning is risky
  • Continuous asset discovery without probing
  • Supplementing active scanning with additional visibility
  • Detecting unauthorized services and shadow IT
  • Real-time vulnerability detection for new connections
ğŸ’œ Hybrid Approach

Most enterprise VM programs use a combination: network scanners for comprehensive coverage of known assets, agents for deep endpoint visibility and roaming devices, and passive scanning for OT environments or continuous discovery.

Authenticated vs. Unauthenticated Scans

The depth of vulnerability detection depends heavily on whether the scanner can authenticate to target systems.

Aspect Unauthenticated Authenticated
Detection Depth Surface-level (banners, exposed services) Deep (installed software, configs, patches)
Accuracy Higher false positives Higher accuracy, fewer false positives
Coverage ~30-40% of vulnerabilities ~90%+ of vulnerabilities
Setup Complexity Simple (no credentials needed) Complex (credential management)
Use Case External scans, discovery, quick assessments Internal scans, compliance, detailed inventory
âš ï¸ Unauthenticated Scan Limitations

Unauthenticated scans miss the majority of vulnerabilities. A scan showing "no vulnerabilities" without credentials is almost certainly incomplete. Always use authenticated scanning for compliance and accurate risk assessment.

Commercial vs. Open Source Scanners

Both commercial and open source scanners have their place in a VM program. The choice depends on budget, requirements, and organizational maturity.

ğŸ’¡ Interview Question

"When would you choose an open source scanner over commercial?"

Sample answer: "Open source scanners work well for specific use cases: Trivy is excellent for container scanning in CI/CD pipelines, Nuclei is powerful for custom web application checks, and OpenVAS can serve small organizations with limited budgets. However, for enterprise VM programs, I'd choose commercial scanners for their broader coverage, faster plugin updates, compliance templates, professional support, and integrated reporting. I often use open source tools to supplement commercial scanners for specific needs."

Scanner Selection Criteria

Selecting a vulnerability scanner requires evaluating multiple factors beyond just detection capabilities.

Technical Requirements

  • Coverage: Does it support all your platforms (Windows, Linux, Mac, network devices, cloud, containers)?
  • Detection Quality: Plugin count, update frequency, false positive rates
  • Authentication: Supported protocols (SMB, SSH, WinRM, SNMP, APIs)
  • Compliance: Built-in compliance checks for PCI DSS, CIS, HIPAA, etc.
  • Scalability: Can it handle your asset count and scan frequency?
  • API: REST API for automation and integration

Operational Requirements

  • Deployment Model: On-premises, cloud, hybrid?
  • Ease of Use: Learning curve, UI quality, documentation
  • Reporting: Built-in reports, customization, export formats
  • Integration: SIEM, ticketing, CMDB, SOAR connectors
  • Support: Vendor support quality, community resources

Business Requirements

  • Budget: License cost, infrastructure costs, FTE requirements
  • Vendor Stability: Market position, financial health, roadmap
  • Compliance: Approved Scanning Vendor (ASV) certification for PCI
  • Contracts: Licensing model (per-asset, per-IP, subscription)
