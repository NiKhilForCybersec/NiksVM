üìã Role Overview

What Does a Vulnerability Analyst Do?

The Vulnerability Analyst is the operational backbone of the vulnerability management program. They are responsible for the day-to-day execution of scanning, triage, prioritization, ticketing, tracking, and reporting. Unlike engineers who build and maintain the infrastructure, analysts focus on interpreting scan results, validating findings, coordinating remediation, and communicating with stakeholders.

Primary Responsibilities

  • Scan Execution & Scheduling: Configure and execute scheduled and ad-hoc vulnerability scans across network segments, cloud environments, and endpoints
  • Results Triage: Review raw scan output, filter noise, identify true positives, and validate vulnerability existence
  • Risk Prioritization: Apply CVSS, EPSS, KEV data, and business context to rank vulnerabilities by actual risk
  • Ticket Management: Create, assign, track, and close remediation tickets in ServiceNow/Jira
  • False Positive Analysis: Investigate and document false positives, create exception rules
  • SLA Monitoring: Track remediation timelines, send escalations for breaches
  • Stakeholder Communication: Lead weekly remediation meetings, provide status updates to management
  • Reporting & Dashboards: Generate executive reports, maintain KPI dashboards, trend analysis

Required Skills

  • Scanner Proficiency: Tenable.io/Nessus, Qualys VMDR, Rapid7 InsightVM, Microsoft Defender VM
  • Ticketing Systems: ServiceNow SecOps, Jira, BMC Remedy
  • Risk Frameworks: CVSS v3.1/v4.0, EPSS, CISA KEV catalog
  • Operating Systems: Windows Server, Linux (RHEL, Ubuntu), network devices
  • Networking Basics: TCP/IP, ports, protocols, firewall concepts
  • Reporting Tools: Power BI, Splunk dashboards, Excel/CSV analysis
  • Soft Skills: Communication, stakeholder management, documentation
  • Certifications: CompTIA Security+, CySA+, CEH, GEVA

‚è∞ Day-to-Day Responsibilities (Hourly Breakdown)

A typical day for a vulnerability analyst involves structured tasks across scanning, triage, communication, and reporting. Here's a detailed hourly breakdown:

8:00 AM - 9:00 AM | Morning Review
Overnight Scan Results Review
  • Check scanner console for completed overnight scans (authenticated scans typically run 2-6 AM)
  • Review scan completion status - identify any failed scans due to credential issues, network timeouts, or target unavailability
  • Check email for critical vulnerability alerts (CVSS 9.0+) that require immediate attention
  • Review CISA KEV updates from previous day - check if any new KEVs affect your environment
  • Check EPSS score updates for existing vulnerabilities - reprioritize if scores increased significantly
  • Review overnight tickets created by automated workflows
9:00 AM - 11:00 AM | Triage & Validation
Primary Vulnerability Triage Session
  • Export new findings: Pull vulnerabilities discovered in past 24 hours from scanner
  • Deduplicate: Group same vulnerability across multiple assets, identify affected asset count
  • Validate criticality: Cross-reference CVSS score with asset criticality tier (Tier 1 = business critical, Tier 2 = important, Tier 3 = standard)
  • Check exploitability: Verify if public exploit exists (ExploitDB, Metasploit modules), check if actively exploited (KEV)
  • False positive check: Review plugin output details, verify software version detected matches actual installed version
  • Asset context: Determine if asset is internet-facing, contains sensitive data (PII/PCI), or is in production vs. dev
  • Assign risk score: Calculate final priority using formula: CVSS + Asset Criticality + Exposure + Exploitability
11:00 AM - 12:00 PM | Ticket Creation
Remediation Ticket Generation
  • Create tickets: Generate ServiceNow/Jira tickets for validated high/critical vulnerabilities
  • Populate required fields: CVE ID, affected assets, CVSS score, recommended fix, SLA due date
  • Assign ownership: Route to appropriate team based on CMDB owner - Infrastructure, App Team, DBA, Network
  • Attach evidence: Include scanner output screenshot, vulnerability description, remediation guidance
  • Set priority: Map vulnerability severity to ticket priority (Critical=P1, High=P2, Medium=P3, Low=P4)
  • Link related tickets: Associate with parent change request if patch deployment planned
12:00 PM - 1:00 PM | Lunch Break
Break
Monitor critical alerts on mobile, respond to urgent Slack/Teams messages if needed.
1:00 PM - 2:30 PM | Stakeholder Communication
Remediation Meetings & Follow-ups
  • Weekly remediation meeting: Review open vulnerability backlog with infrastructure and application teams
  • Discuss blockers: Identify vulnerabilities stuck due to application dependencies, change freeze, resource constraints
  • Negotiate timelines: Work with asset owners on realistic remediation dates within SLA
  • Exception requests: Review requests for risk acceptance, ensure proper justification and compensating controls
  • Escalation decisions: Flag vulnerabilities approaching SLA breach to management
  • Send status emails: Provide weekly summary to business unit leaders on their vulnerability posture
2:30 PM - 4:00 PM | Verification & Closure
Remediation Verification Scans
  • Trigger verification scans: Run targeted rescans on assets where patches were applied
  • Compare results: Verify vulnerability no longer detected after remediation
  • Close tickets: Update ticket status to "Resolved" with verification evidence
  • Handle failures: If vulnerability persists, reopen ticket, engage with remediation team
  • Update metrics: Record MTTR (Mean Time to Remediate), track against SLA targets
  • Document exceptions: For vulns that cannot be patched, ensure risk acceptance is documented with expiration date
4:00 PM - 5:00 PM | Reporting & Planning
Dashboard Updates & Next Day Prep
  • Update dashboards: Refresh Power BI/Splunk dashboards with latest vulnerability counts
  • Generate reports: Create daily/weekly reports for management showing remediation progress
  • Trend analysis: Review vulnerability aging report - identify vulns open > 30/60/90 days
  • Schedule scans: Configure next day's scan jobs, verify credential validity
  • Review threat intel: Check for new critical CVEs published that may affect environment
  • Knowledge base: Document any new false positive patterns or troubleshooting procedures discovered

‚ö†Ô∏è Common Discrepancies & Issues

Analysts frequently encounter these discrepancies between scan results and reality. Understanding these helps avoid wasted remediation effort:

üî¥ False Positive: Backported Security Patches (Linux)
Scanner reports Apache 2.4.6 vulnerable to CVE-2023-XXXX, but RHEL/CentOS backports security fixes without changing version number. Scanner sees "old" version and flags it, but the actual vulnerability was patched.
Resolution: Run rpm -q --changelog httpd | grep CVE-2023-XXXX to verify if patch was backported. Use authenticated scans that check package manager metadata, not just banner versions. Mark as false positive with evidence and create scanner exception rule for this CVE+package combination.
üî¥ False Positive: Uninstalled Software Remnants
Scanner detects vulnerable Java 8u202 on Windows server, but Java was uninstalled months ago. Registry entries, environment variables, or leftover files in Program Files trigger the detection.
Resolution: Verify via java -version command and check Add/Remove Programs. If truly uninstalled, clean up remnants: remove JAVA_HOME environment variable, delete leftover folders, clean registry keys. Document as false positive and request rescan.
üü° Scan Credential Failure - Incomplete Results
Scan completes but shows 0 vulnerabilities on servers that should have findings. Authenticated scan credentials failed silently, causing scanner to only perform unauthenticated checks which miss most vulnerabilities.
Resolution: Check scan logs for authentication errors. Verify service account password hasn't expired. Confirm account has local admin rights (Windows) or sudo privileges (Linux). Check if firewall blocked WMI/SSH ports. Re-run scan after fixing credentials. Always verify "credentialed checks: yes" in scan results.
üü° Asset Not in CMDB - No Owner Assignment
Scanner discovers vulnerable asset but it's not in the Configuration Management Database. Cannot determine owner for ticket assignment. Often shadow IT, temporary test systems, or missed during onboarding.
Resolution: Query Active Directory for computer object details. Check DHCP/DNS records for hostname patterns indicating ownership. Use network traffic analysis to identify usage patterns. Escalate to IT Asset Management for CMDB update. Once owner identified, ensure CMDB is updated to prevent recurrence.
üü° Duplicate Vulnerabilities Across Scanners
Running multiple scanners (e.g., Tenable for network, Qualys for cloud, Microsoft Defender for endpoints) creates duplicate findings for same vulnerability on same asset with different formats and IDs.
Resolution: Use vulnerability aggregation tool (DefectDojo, Nucleus, Kenna) to deduplicate findings based on CVE + Asset. Establish primary scanner per asset type to avoid overlap. Create master ticket and link scanner-specific findings. Standardize on CVE as common identifier across tools.
üîµ Vulnerability Verified But Patch Breaks Application
Remediation team confirms vulnerability exists and applies patch. Application stops working because patch changes behavior that application depends on. Common with Java, .NET framework, and database patches.
Resolution: Always test patches in non-production environment first. Work with application team to identify compatibility issues. Options: request vendor hotfix, implement compensating controls (WAF rules, network segmentation), accept risk with documentation, or plan application upgrade. Never rollback patches without security team approval.
üîµ CVSS Score Mismatch Between Sources
NVD shows CVSS 7.5, vendor advisory says 8.1, scanner shows 6.8. Different scoring by different entities creates confusion about true severity and SLA assignment.
Resolution: Prefer NVD as authoritative source for base score. Understand that vendor scores may include environmental factors. Scanner scores may be adjusted based on detection confidence. Document your organization's policy for score source. Consider EPSS and KEV status as additional prioritization factors beyond CVSS alone.
üîµ Vulnerability Fixed But Still Appears in Scan
Patch was applied and verified, but next scan still shows vulnerability. Causes: scan cache, incomplete patch installation, multiple instances of software, or scanner plugin lag.
Resolution: Wait for scanner plugin update if CVE is newly patched. Clear scanner cache and force fresh credential authentication. Verify ALL instances of software were patched (e.g., multiple Java installations). Check if patch requires reboot to take effect. Run targeted rescan of specific asset with fresh scan policy.
üî¥ Zero-Day Announced - No Scanner Detection Yet
Critical zero-day vulnerability announced in news/vendor advisory, but scanner doesn't detect it yet. Plugin development can take 24-72 hours for new CVEs.
Resolution: Don't wait for scanner - manually identify affected systems using software inventory. Query CMDB for systems running vulnerable software/version. Use vendor-provided detection scripts or IOC checks. Create tracking ticket manually. Apply patches or workarounds immediately for critical issues. Scanner will catch any missed systems once plugin released.

What Does a Vulnerability Engineer Do?

The Vulnerability Engineer builds, maintains, and optimizes the technical infrastructure that enables the vulnerability management program. While analysts focus on operational tasks, engineers ensure the scanners work correctly, integrations flow data properly, automation reduces manual effort, and the platform scales with organizational growth.

Primary Responsibilities

  • Scanner Architecture: Design and deploy vulnerability scanner infrastructure - on-prem appliances, cloud connectors, agent distribution
  • Credential Management: Configure and maintain scan credentials in vault (CyberArk, HashiCorp), manage service accounts
  • CMDB Integration: Sync asset data from Active Directory, Azure, VMware, ServiceNow CMDB to scanners
  • Ticketing Integration: Build API workflows to auto-create tickets from scan results in ServiceNow/Jira
  • Automation Development: Write Python/PowerShell scripts for data enrichment, reporting, remediation workflows
  • Patch Orchestration: Integrate with SCCM/Intune for patch deployment visibility and coordination
  • Platform Maintenance: Upgrade scanners, apply security patches, monitor performance, troubleshoot issues
  • New Tool Evaluation: POC new scanner technologies, evaluate cloud security tools, recommend improvements

Required Skills

  • Programming: Python, PowerShell, Bash scripting for automation
  • APIs: REST API integration, JSON/XML parsing, authentication methods
  • Scanners (Admin): Tenable.io/SC administration, Qualys backend, Rapid7 console management
  • Infrastructure: Linux server admin, Windows Server, virtualization (VMware, Hyper-V)
  • Cloud Platforms: AWS, Azure, GCP security and API access
  • Networking: Firewall rules, VLANs, routing for scanner placement
  • Databases: SQL for reporting queries, data warehouse concepts
  • Certifications: Tenable Certified Security Engineer, Qualys Certified Specialist, CISSP, GIAC

‚ö†Ô∏è Common Discrepancies & Issues

üî¥ Credential Scan Fails - Access Denied
Authenticated scans failing with "Access Denied" or "Permission Denied" errors. Service account credentials are correct but scan cannot execute privileged checks.
Resolution:
Windows: Verify account is in local Administrators group, WMI access enabled, Remote Registry service running, firewall allows WMI (TCP 135, 445, dynamic ports), UAC remote restrictions disabled for domain accounts.
Linux: Verify sudo NOPASSWD configured for scan account, SSH key authentication working, /etc/sudoers allows required commands, SELinux not blocking access.
üî¥ Scanner Plugin Out of Date - Missing New CVEs
New critical vulnerability announced but scanner not detecting it. Plugin feed is days or weeks behind, missing detection capability for recently published CVEs.
Resolution: Check plugin feed update status in scanner console. Verify internet connectivity for plugin downloads. Force plugin update manually if auto-update stuck. Check vendor status page for plugin release schedule. For zero-days, use manual detection methods while waiting for plugin. Consider supplementing primary scanner with Nuclei templates for rapid CVE coverage.
üü° CMDB Asset Count Mismatch with Scanner
CMDB shows 10,000 servers but scanner only sees 8,500 assets. 1,500 assets missing from vulnerability coverage. Creates blind spots in security posture.
Resolution: Export asset lists from both systems and compare. Common causes: network segmentation blocking scanner, assets powered off, IP address changes, new network segments not added to scan targets. Work with network team to ensure scanner can reach all subnets. Implement continuous asset discovery scanning. Set up CMDB sync to auto-add new assets to scanner.
üü° Scan Timeout on Large/Complex Systems
Scans of database servers, file servers, or systems with many installed applications timeout before completion. Results are partial, missing vulnerabilities.
Resolution: Increase per-host scan timeout settings (default often 30-60 minutes, increase to 2-4 hours for complex systems). Reduce concurrent checks per host. Consider agent-based scanning for complex hosts - agents can scan incrementally without timeout pressure. Exclude non-essential checks (compliance audits) from performance-sensitive scans.
üîµ API Rate Limiting Breaking Automation
Automated scripts hitting API rate limits (HTTP 429 errors). Bulk operations like ticket creation or asset updates failing mid-execution.
Resolution: Implement exponential backoff in scripts - wait and retry on rate limit errors. Add delays between API calls (100-500ms). Batch operations where possible (bulk ticket creation vs. individual). Cache frequently accessed data to reduce API calls. Request rate limit increase from vendor if business-justified. Monitor API usage dashboards to stay under limits.
üîµ Multi-Cloud Visibility Gaps
Organization uses AWS, Azure, and GCP but scanner only has visibility into AWS. Cloud teams provisioning workloads that never get scanned.
Resolution: Configure cloud connectors for each cloud provider. Use native tools where appropriate (AWS Inspector, Azure Defender, GCP Security Command Center) and aggregate findings. Implement cloud security policy requiring vulnerability scanning before workload deployment. Consider CSPM tools for multi-cloud posture management. Automate connector setup as part of cloud account provisioning.

1. Asset Discovery

You cannot protect what you cannot see. Asset discovery establishes complete visibility of the attack surface.

  • Network scanning: Discover live hosts via ping/ARP sweeps
  • CMDB integration: Sync from ServiceNow, Active Directory
  • Cloud connectors: AWS, Azure, GCP asset APIs
  • Agent deployment: Endpoint agents for off-network devices
  • Shadow IT detection: Find unmanaged/unauthorized assets

2. Vulnerability Scanning

Systematic identification of security weaknesses across all asset types.

  • Authenticated scans: Credentialed checks for complete visibility
  • Network scans: TCP/UDP port scanning and service detection
  • Agent-based: Continuous assessment without network dependency
  • Web app scans: OWASP Top 10, SQL injection, XSS
  • Container scans: Image vulnerabilities in registries/runtime
  • Cloud config: Misconfigurations in IaaS/PaaS/SaaS

3. Analysis & Prioritization

Not all vulnerabilities are equal. Risk-based prioritization focuses effort on what matters.

  • CVSS scoring: Industry standard severity rating (0-10)
  • EPSS: Probability of exploitation in next 30 days
  • CISA KEV: Known actively exploited vulnerabilities
  • Asset criticality: Business impact of affected system
  • Exposure: Internet-facing vs. internal only
  • False positive triage: Validate findings before creating work

4. Remediation

Fixing vulnerabilities through patching, configuration, or compensating controls.

  • Patching: Apply vendor security updates (SCCM, Intune, Ansible)
  • Configuration: Harden settings, disable vulnerable features
  • Upgrade: Move to supported software versions
  • Compensating controls: WAF rules, network segmentation when patching impossible
  • Risk acceptance: Documented business decision to accept residual risk

5. Verification

Confirming remediation was successful before closing tickets.

  • Rescan: Targeted scan of remediated assets
  • Manual validation: Verify patch installed, config changed
  • Penetration testing: Confirm exploit no longer works
  • Continuous monitoring: Ensure vulnerability doesn't reappear

6. Reporting

Communicating risk posture and program effectiveness to stakeholders.

  • Executive dashboards: High-level risk trends for leadership
  • Technical reports: Detailed findings for remediation teams
  • SLA tracking: Remediation timeline compliance
  • Compliance evidence: Audit documentation for PCI, SOX, HIPAA
  • KPIs: MTTR, aging, coverage metrics

CVSS - Common Vulnerability Scoring System

Industry-standard severity scoring maintained by FIRST.org. Current version: CVSS v4.0

ScoreSeverityTypical SLA
9.0 - 10.0Critical7 days
7.0 - 8.9High30 days
4.0 - 6.9Medium90 days
0.1 - 3.9Low180 days

Limitation: Static score doesn't reflect real-world exploitation likelihood

EPSS - Exploit Prediction Scoring System

Machine learning model predicting probability of exploitation in next 30 days. Updated daily.

  • Score range: 0.0 to 1.0 (0% to 100% probability)
  • High risk threshold: EPSS > 0.1 (10%) deserves attention
  • Critical threshold: EPSS > 0.5 (50%) - prioritize immediately
  • Data sources: Exploit code availability, threat intel, vulnerability age

Best use: Combine with CVSS - High CVSS + High EPSS = Highest Priority

CISA KEV - Known Exploited Vulnerabilities

CISA catalog of vulnerabilities confirmed actively exploited in the wild. Mandatory remediation for US federal agencies.

  • Binary indicator: Either on the list or not
  • Action: If on KEV, prioritize regardless of CVSS score
  • Updates: Multiple times per week as new exploitation observed
  • SLA override: KEV vulnerabilities often get expedited 24-48 hour SLAs

URL: cisa.gov/known-exploited-vulnerabilities-catalog
